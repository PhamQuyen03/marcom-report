Mô hình only slug cho report HTML

Public endpoint của mỗi report chỉ còn một slug ổn định theo dạng /reports/<slug>. Không còn version path và không còn latest. Rollback được xử lý bằng artifact cũ của chính slug đó.

Public URL

/reports/<slug>

Artifact

Một artifact rõ ràng cho mỗi slug.

Rollback

Redeploy artifact cũ của cùng slug.

Notify

Success và failure đều gửi Teams.

Nguyên tắc CI/CD chung

  • Build một lần, deploy một artifact rõ ràng cho từng slug.
  • Không build lại trong production stage.
  • Validate sớm file slug, route public và smoke check.
  • Rollback bằng artifact cũ của chính slug đó.
  • Success hoặc failure đều notify về Microsoft Teams.

Luồng pipeline chuẩn

  1. Detect thay đổi ở report source.
  2. Validate naming rule của slug.
  3. Build artifact hoặc build image.
  4. Deploy lên môi trường mục tiêu.
  5. Smoke check /reports/<slug>.
  6. Notify Microsoft Teams.
  7. Nếu cần rollback thì redeploy artifact cũ của slug đó.

Giải pháp 1: AKS

Đây là giải pháp phù hợp nhất với yêu cầu hiện tại khi repo chính vẫn là app Next.js và công ty đã có sẵn platform Kubernetes, ACR, Azure DevOps và team DevOps vận hành.

Checklist

  • Build image Next.js.
  • Push image lên ACR.
  • Deploy lên AKS.
  • Expose qua Gateway API publish layer.
  • Smoke check /reports/<slug>.
  • Notify Teams.

Giải pháp 2: Cloudflare Pages

Phù hợp khi muốn edge hosting global và deploy nhanh từ Azure DevOps, nhưng không phải lựa chọn đồng bộ nhất với production stack hiện tại.

Giải pháp 3: Vercel

Phù hợp khi team quen hệ sinh thái Vercel và cần managed frontend hosting với public route theo slug.

Giải pháp 4: VM với Nginx

Phù hợp khi muốn toàn quyền với routing và file system, nhưng ít đồng bộ hơn với production stack hiện tại.

Khuyến nghị thực tế hiện tại

Với bối cảnh hiện tại, hướng nên chốt để đồng bộ nhất là AKS. Các giải pháp còn lại nên coi là phương án thay thế theo governance, chi phí hoặc mục đích riêng.
